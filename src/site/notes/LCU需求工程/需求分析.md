需求分析 需求的分类分为功能型 非功能形的有什么区别？ 对于功能型的，如何分析？用什么手段？优缺点？ 对于非功能性的 质量？安全？性能？
### 功能性需求 v.s. 非功能性需求
功能性需求规定了系统应该做什么，描述了系统必须为用户提供的服务、任务或具体功能
非功能性需求规定了应该怎么样，它描述了系统提供功能时的质量属性、标准和约束。
比如对于图书馆平台：具体的功能就是功能性需求 比如用户可以根据书名、作者搜索图书；一些性能指标就是非功能性需求，比如图书搜索的响应时间必须在2秒以内

### 功能性需求分析
一般有两种方式，结构化分析和面向对象分析
结构化分析：一种经典的，自顶向下的分析方法。主要使用**数据流图 (DFD)** 来描述数据在系统中如何流动和被处理，配合**数据字典 (DD)** 来定义数据。以及**加工逻辑说明(如伪代码、判定表)** 来描述具体的业务规则
优点：1) 逻辑清晰，能够直观的展示数据处理的逻辑和流程 2) 易于理解：对数据密集型的业务(如信息管理系统)，有效且易于沟通
缺点：1）分析与设计脱节：与面向对象设计存在鸿沟 2）当需求发生更改时，修改大量关联的DFD图会非常困难

面向对象分析
这是现代软件开发的主流方法。主要使用**统一建模语言 (UML)** 中的图来进行分析和建模，核心手段包括：
- **用例图 (Use Case Diagram):** 从用户视角描述“谁（参与者）能用系统做什么（用例）”，是功能性需求分析的起点。
- **类图 (Class Diagram):** 识别系统中的核心业务实体（类）及其属性、方法和相互关系，构建系统的静态结构模型。
- **活动图/顺序图 (Activity/Sequence Diagram):** 描述一个用例的具体执行流程或对象间的交互顺序，将功能细化。

优点：
- **模型一致性：** 分析、设计、编码阶段使用一致的模型，减少了沟通和转换成本。
- **更贴近现实：** 能更好地模拟现实世界中的事物及其关系，易于理解和维护。
- **复用性好：** 封装、继承等特性使得模型和代码的复用性更高。

**缺点：**
- **学习曲线较陡：** 掌握整套UML和OO思想需要一定的学习成本。
- **对于简单项目可能过重：** 对于一些非常简单的项目，使用OOA可能显得“杀鸡用牛刀”

### 非功能性需求分析
主要问题就是分析什么？
性能：关注系统的响应速度，吞吐量，并发用户数和资源利用率
- **响应时间：** “95%的查询请求必须在3秒内返回结果。”
- **吞吐量：** “系统必须能支持每秒处理100笔交易。”
- **并发用户数：** “系统必须能同时支持500个用户在线操作而不出现性能显著下降。”
- 安全性
	- **认证与授权：** “用户必须通过用户名和密码登录”，“管理员角色才能访问后台管理功能”。
	- **数据加密：** “所有在网络上传输的敏感数据必须使用HTTPS加密”，“用户密码必须经过哈希加盐处理后存储”。
	- **审计与日志：** “所有对关键数据的修改操作都必须记录日志”。
- 质量
	- 可靠性 “系统的平均无故障时间（MTBF）必须达到99.9%”
	- 可用性 未经培训的用户应能在10分钟内独立完成核心操作
	- 可维护性 “代码必须遵循XX编码规范”、“核心模块必须有清晰的文档”。

### 校园二手交易平台
> [!Question]
> 校园二手交易平台 请详细阐述你对该应用的软件需求规划，并重点突出至少三个创新功能点及其设计思路与预期用户价值。（要求：以项目文档形式呈现，内容结构清晰，功能点描述详细）

**需求规划**
用户与认证模块
- 基本需求
	- 注册: 支持手机号注册。为确保平台安全与纯净，强制要求用户进行学号与姓名绑定，完成在校生身份认证后，方可进行发布商品和购买操作
	- 登录: 支持手机验证码登录、密码登录
	- 个人主页: 用户可以编辑昵称、头像。系统自动展示用户的认证状态, 信用积分、发布的商品、收到的评价等
- 业务流程：新用户通过手机号注册后，进入身份认证流程，提交学号与姓名。系统与学校数据库接口（或后台人工审核）进行验证，通过后账户被激活，用户即可正常使用所有交易功能。

商品与浏览模块
- 基本需求：
	- 商品发布：用户可以方便地上传商品图片（支持多图）、填写商品标题、描述、选择分类、设定价格、标记新旧程度和交易地点偏好
	- 商品分类：提供符合校园特色的分类导航，如“教材书籍”、“电子数码”、“宿舍神器”、“运动户外”、“美妆服饰”等。
	- 商品搜索与筛选：支持关键词搜索。提供多维度筛选功能，如按分类、价格区间、新旧程度、发布时间、校区等进行筛选。
	- 商品详情：展示商品的所有信息、卖家信息、留言/评论区
- 业务流程：用户点击发布按钮，按引导填写信息并提交。系统审核通过后，商品进入信息流。其他用户可通过浏览分类、搜索、筛选等方式发现商品，并点击进入详情页查看

沟通与交易模块
- 基本需求：
	- 内置聊天系统:支持买卖双方在应用内进行安全的文字、图片沟通，商议商品细节和交易事宜。聊天记录将作为交易纠纷的凭证。
	- 下单与状态管理:买家可以对商品发起“我想要”的请求，卖家确认后，商品状态变为“交易中”。
	- 线下交易确认：双方完成线下见面交易后，由买家在App内进行“确认收货”操作，卖家确认后，交易完成
	- 评价体系：交易完成后，买卖双方可以进行双向互评和打分，评价内容将计入用户的信用体系
- 业务流程：买家通过聊天与卖家沟通 -> 双方约定交易 -> 买家发起购买请求 -> 卖家确认 -> 双方线下交易 -> 买卖双方在App内确认完成 -> 双方互评

**创新功能点**
创新点1：基于校园实名认证的信用体系
- 设计思路：传统的星级评价容易被刷，且无法有效约束用户行为。本平台将强制性的**“在校生实名认证”作为基石，构建一个多维度的信用积分体系。用户的初始信用分由认证身份赋予，后续通过以下行为进行动态增减
	- 加分项：每完成一笔成功的交易、获得一次好评、快速响应买家消息、积极举报违规内容
	- 减分项：获得差评、被多人举报并查实（如放鸽子、货不对板）、取消交易次数过多
- 预期用户价值：
	- 极大提高安全性
	- 建立信任闭环

创新点2：校园角安全交易点与地图导航
- 设计思路：在哪儿见面”是校园线下交易的一大难题，既要方便又要安全。本功能与学校安保部门合作，在校园地图上预设多个官方认证的**“安全交易角”，通常位于图书馆门口、教学楼大厅、食堂等人流量大且有监控覆盖的区域
	- 买卖双方在聊天中，可以一键分享并选择某个“校园角”作为交易地点
	- App内嵌校园地图，用户选定地点后，可直接为双方生成导航路线
- 预期用户价值
	- 保障人身安全
	- 提升交易便捷性：省去了双方反复商议地点的麻烦

创新点3：课表关联与“毕业季/开学季”一键清仓/采购模式
- 设计思路：精准匹配需求，并解决校园周期性的爆发式交易需求
	- 课程表关联： 允许用户导入（或手动输入）自己的课程表。当卖家发布教材类书籍时，可以关联具体课程代码（如CS101）。系统会自动将这些书籍精准推送给所有选了这门课的同学。
	- 周期性模式： 在每年6月的“毕业季”和9月的“开学季”，App将自动切换到特殊模式。
    - 毕业季（清仓模式）： 鼓励毕业生用户创建“我的宿舍好物清单”，打包出售宿舍内的各类物品（如台灯、风扇、收纳盒），并允许其他用户“一键预约”整包购买。
    - 开学季（采购模式）： 为新生智能生成“宿舍必备物品”采购清单，并自动匹配平台上的相关二手商品
- 预期用户价值：
	- 精准匹配 ： 人找书 变为 书找人
	- 解决周期性刚需

	- 用户与认证

	- 图书详细与状态：展示图书的完整信息（封面、简介、目录、豆瓣评分等），并实时、清晰地显示该书在各个馆藏地的具体位置（如“主馆-社科阅览区-K89/23”）、在架状态、可借复本数及应还日期

		- 电子借阅证

	- 统计与报表
- 业务流程：图书管理员通过Web后台登录，根据其权限执行相应的管理操作，如图书编目、订单处理、读者信息维护等，并通过数据报表监控图书馆运营状况。

- 用户价值：效率 提高体验

创新2：AI智能推荐
- 设计思路：分析借阅历史，提供猜你喜欢AI深度推荐功能
